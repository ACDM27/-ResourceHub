一句话总结

这篇文章的核心观点不是“多智能体一定更强”，而是：当任务处在模型能力边界附近时，合理的 harness 能显著提高完成质量；但随着模型变强，旧的 scaffold 会变成负担，所以 harness 必须持续被简化、重构、重估。

文章要解决什么

Anthropic 想解决两个看似不同、实则相关的问题：一是让 Claude 产出更有审美和原创性的前端设计；二是让 Claude 在几小时级别的长时间自主开发中，持续稳定地构建完整应用。

作者的关键发现

单靠“一个 agent 从头干到尾”很容易在长任务里失去连贯性，也很不擅长评价自己。把“生成”和“评估”拆开，再配合结构化交接和可执行的检查标准，质量会明显提升。

真正重要的结论

Harness 不是固定模板，而是一套随着模型能力移动的“支架系统”。模型提升后，原本必要的结构可能不再必要；但新的更高难任务，也会催生新的 harness 组合。

为什么朴素方案会失效

文中先解释了为什么“直接让一个模型长时间工作”经常不靠谱，主要有两个问题。

1. 长任务中的上下文退化

随着上下文窗口越来越满，模型容易丢失连贯性，甚至出现作者所说的 “context anxiety”： 明明工作还没做好，却因为担心上下文快到极限而提前收尾。早期方案通过 context reset 来解决，让新 agent 继承结构化交接文档继续干活。

2. 自我评价天然偏宽松

让模型评估自己时，它往往会过度乐观，尤其在设计这类主观任务上更明显。即使是有可验证结果的软件任务，模型也常常会发现问题后又“说服自己问题不大”，导致 QA 形同虚设。

文章最重要的思路转折在这里：与其逼生成器学会严厉地否定自己，不如单独训练一个更怀疑、更挑剔的评估器。这样生成器才能获得真正可执行的外部反馈。

前端设计实验：把主观审美变成可评分对象

作者先在前端设计上做实验，因为这里最容易暴露“模型自评太宽松”的问题。默认情况下，Claude 会做出安全、可用、但不出彩的页面。

Generator 先产出页面

根据用户提示生成 HTML / CSS / JS 前端界面。它不仅要实现功能，还要尝试建立明确的视觉风格和情绪。

Evaluator 进入页面打分

评估器通过 Playwright 直接操作真实页面，而不是只看静态截图。它会浏览、截图、检查，再按标准给出细颗粒度反馈。

多轮反馈驱动风格进化

生成器根据反馈决定继续打磨当前方向，还是彻底换一种美学路线。一个完整循环通常要跑 5 到 15 轮，最长可达 4 小时。

四个评分标准

  • 设计质量：整体是否统一，是否形成独特气质，而不是零碎拼贴。
  • 原创性：有没有真实设计决策，还是在复用模板、库默认样式和典型 AI 套路。
  • 工艺：排版、间距、配色、对比度这些基础是否扎实。
  • 功能性：用户是否能看懂界面、找到主操作、顺利完成任务。

为什么这组标准有效

作者刻意把“设计质量”和“原创性”权重拉高，因为模型天生在功能和工艺上已经不差，真正缺的是审美风险和个性化表达。换句话说，评分机制本身就在把模型从“AI 套壳感”往“更像设计师作品”推。

早期输出
页面通常功能正确，但视觉保守、套路明显，像“合格模板”而不是“有记忆点的设计”。
中期迭代
评估器开始逼生成器改掉泛化套路，页面逐渐出现更明确的视觉语言和更大胆的结构。
代表性突破
在“荷兰艺术博物馆”案例中，模型从常规深色落地页，突然跃迁到一个用 CSS 透视做出的 3D 展厅空间，这是作者此前单次生成里没见过的创造性跳跃。

扩展到全栈开发：从设计打分到软件工程流程

在验证“生成器 + 评估器”模式有效后，作者把它迁移到完整应用开发场景，并形成了更成熟的三代理结构。

Planner

接收 1 到 4 句简短需求，把它扩展成完整产品规格。作者刻意让 Planner 停留在产品和高层技术设计层面，不提前写死底层实现，以免错误设计向后级联。

Generator

按 feature 或 sprint 实现应用，使用 React、Vite、FastAPI、SQLite / PostgreSQL 等技术栈，并在每轮结束后先做一次自检，再提交给 QA。

Evaluator

用 Playwright 像真实用户一样点击应用，检查 UI、API、数据库状态，再按照功能深度、视觉设计、代码质量等标准打分。任何一项低于阈值，该轮就算失败。

文中一个很值得借鉴的细节是 “sprint contract”。每次开发前，生成器和评估器要先协商这一轮到底交付什么、如何验证算完成。这个动作把模糊规格转成可测试契约，极大降低了“做了很多但做偏了”的风险。
文件作为代理之间的通信介质 先定完成标准，再开始写代码 QA 不只看界面，还查 API 和数据库 结构化交接比单纯压缩对话更稳

案例与成本对比

文章用了两个案例来证明 harness 的收益，同时也很诚实地展示了它的代价：时间长、成本高、而且不是所有任务都值得这么做。

案例一：复古游戏制作器 RetroForge

方案 时长 成本 结果
单 Agent 20 分钟 9 美元 表面上像能用，但真正试玩时核心玩法不工作，实体出现了却无法响应输入。
完整 Harness 6 小时 200 美元 功能范围更大、界面更完整、游戏可实际运行，还集成了 AI 生成内容能力，但仍存在流程引导和边角体验问题。

Harness 在这里提升了什么

  • Planner 把一句话需求扩成了 16 个功能点、10 个 sprint。
  • Evaluator 在合同级别追踪验收条件，能给出非常具体的 bug 定位。
  • 最终成品不只是“看起来像应用”，而是核心交互真的能跑起来。

Evaluator 抓到的典型问题

  • 矩形填充工具实际上没有填满矩形区域。
  • 删除实体的条件判断写错，导致点选后删不掉。
  • FastAPI 路由顺序导致重排接口被错误匹配，返回 422。

案例二：浏览器 DAW（数字音频工作站）

阶段 时长 成本
Planner 4.7 分钟 0.46 美元
Build Round 1 2 小时 7 分钟 71.08 美元
QA Round 1 8.8 分钟 3.24 美元
Build Round 2 1 小时 2 分钟 36.89 美元
QA Round 2 6.8 分钟 3.09 美元
Build Round 3 10.9 分钟 5.88 美元
QA Round 3 9.6 分钟 4.06 美元
总计 3 小时 50 分钟 124.70 美元

这个阶段有一个很关键的变化：随着 Opus 4.6 变强，作者开始移除原本的 sprint 结构，让生成器一次性跑更长时间，只在末尾加入 QA 回合。结论并不是“Evaluator 永远必要”，而是： 当任务刚好落在模型单独完成的能力边界之外时，Evaluator 的投入很值；如果任务已经被模型稳稳覆盖，那么它就会变成额外开销。

我的分析与启发

1. 这篇文章真正谈的是“外部化判断”

很多人读这类文章会把重点放在多 agent 数量上，但更核心的其实是把“判断标准”和“验收逻辑”从生成过程里剥离出来。模型可以负责创造，但不能同时担任自己最苛刻的审稿人。

2. Contract 是工程落地的关键桥梁

从模糊产品需求到可验证实现，中间最容易断层。作者用 sprint contract 填上了这道缝，这比单纯给一个更长的 prompt 更像真正的软件工程流程。

3. 评分标准会反过来塑造输出风格

文中提到一句类似“最好的设计像博物馆级别作品”的措辞，会让页面朝某类审美收敛。这提醒我们：评估体系不是中性的，它本身就携带价值观，会影响生成结果的分布。

4. Harness 不是越复杂越好

作者在后半段反复强调，任何一个结构组件都只是对“模型当前做不到什么”的假设。一旦模型变强，这个组件就要重新接受审查。能删就删，删不掉的才是 load-bearing。

适合借鉴到实际项目的四条做法

  • 先写出可评分标准，不要只写“做得更好看”“做得更完整”。
  • 让生成和验收分离，尤其在设计、产品和 QA 环节。
  • 把交接材料写成结构化文件，而不是依赖长对话记忆。
  • 每次模型升级后，都要重新测试哪些 harness 组件还值不值保留。

这套方法的现实边界

  • 成本高，不适合低价值、短周期、低风险任务。
  • 评估器也会犯错，需要持续用真实失败案例校准。
  • 对“品味”“常识”“听感”这类主观能力，模型仍有明显上限。
  • 如果验收指标定义不对，系统会朝错误方向稳定优化。
